## P0 — Breaks Functionality (Fix Immediately)

### P0-1 — `/keep` command crashes with KeyError
**File**: `telegram/upload_handlers.py:332`
When a duplicate is detected during upload, `_pending_duplicates[chat_id]` is populated without `dest_path`. Then `/keep` does `dup["dest_path"]` → `KeyError`. Users cannot save files when duplicates are detected.
**Fix**: Store `dest_path` when populating `_pending_duplicates`, computed from `choice` + `pending.filename`.

### P0-2 — `/empty_trash` crashes with ImportError
**File**: `telegram/upload_handlers.py:433`
`from . import store as _st` imports from the `telegram` subpackage which has no `store.py`. Should be `from .. import store as _st`. Crashes at runtime when any user presses the "Empty Trash" inline button.
**Fix**: Change to `from .. import store as _st`.

## P1 — High Risk (Fix This Week)

### P1-1 — NameError in evening duplicate report when bot is inactive
**File**: `main.py:108`
`_send_evening_duplicate_report()` calls `_get_linked_ids()` only if `_tb_mod._application is not None`, but then references `linked` unconditionally in the log statement. If bot is inactive, `linked` is undefined → `NameError`.

### P1-2 — Non-atomic rate limiting — bypassable via concurrency
**File**: `telegram/bot_core.py:99–112`
`_is_rate_limited()` reads, modifies, and writes `_chat_timestamps[chat_id]` non-atomically. Two concurrent requests from same chat can both pass the limit check simultaneously. Fix: use a lock per chat_id or restructure to mutate in place.

### P1-3 — Uncommitted pHash insert in single-file check
**File**: `duplicate_scanner.py:327–337`
`_check_single_sync()` inserts a new file into `phash_cache`, then closes the connection without an explicit commit. SQLite in WAL mode will eventually auto-commit but this is not guaranteed. Newly uploaded images won't appear in subsequent incremental checks until the next full scan.
**Fix**: Add `conn.commit()` before `conn.close()`.

### P1-4 — pHash connection leak on exception
**File**: `duplicate_scanner.py:264`
In `_scan_sync_similar_files()`, `conn = _phash_conn()` then `conn.close()` is called at end of function, but if an exception occurs during the union-find loop (e.g., OOM), the connection is leaked. Wrap in `try/finally` or use a context manager.

### P1-5 — `/empty_trash` leaks exception details to user
**File**: `telegram/upload_handlers.py:343`
Raw exception messages from `except Exception as exc` are sent directly to Telegram chat. May leak internal paths, stack frames, or sensitive state. Log the error server-side, send a generic message to user.

### P1-6 — Document index path-LIKE query runs on multi-word input
**File**: `document_index.py:289`
The supplemental `LIKE /personal/{word}/%` query runs for single-alpha words but `word` is the full query string which can be multi-word after FTS tokenisation. While there's a `word.isalpha()` guard, `word` is split from the raw query inconsistently.
**Fix**: Ensure the `word` variable used in LIKE query is always a single clean token.

## P2 — Important (Fix Next Sprint)

### P2-1 — `shared` path still in config but folder deleted
**File**: `config.py:133,194`
`shared_dir` and `shared_path` still defined in config after the `shared/` folder was removed from the NAS. Nothing references `shared_path` in app code currently, but it's a stale abstraction that could confuse future developers.

### P2-2 — Unbounded in-memory dicts (potential DoS)
**File**: `telegram/bot_core.py:48,70,89`
`_last_results`, `_pending_uploads`, `_pending_duplicates`, `_chat_timestamps` grow unboundedly. `_cleanup_pending_uploads()` is only called in `_handle_media_message`. A user spamming searches with many chat IDs could exhaust RAM on the 8 GB Radxa.
**Fix**: Add max-size eviction (already done for `_last_results` via `_PENDING_MAX_ENTRIES` — apply same pattern to others).

### P2-3 — `uuid` imported but unused in auth.py
**File**: `auth.py:16`
`import uuid` is never used. Remove it.

### P2-4 — `psutil` used without None-guard in /status handler
**File**: `telegram/search_handlers.py`
`psutil` is imported with a try/except fallback to `None`, but the `/status` handler calls `psutil.cpu_percent()` without checking if `psutil is None`. Crashes on systems without psutil.

### P2-5 — Duplicate scan is O(n²) union-find — no timeout
**File**: `duplicate_scanner.py:272–278`
The union-find loop is O(n²) in the number of images. With 10,000 images this is 50M Hamming comparisons. On ARM at ~100M ops/sec this could take 30+ seconds. No timeout or progress reporting. Consider blocking the scan at 5,000 images or adding a configurable limit.

### P2-6 — SHA-256 scan runs per-root with no cross-root deduplication in exact_all
**File**: `duplicate_scanner.py` (`_exact_all` function in scanner)
The merge logic in `_exact_all()` correctly merges sets by hash across roots, but `_scan_sync()` itself logs "files_found" per root and is called per root — so a file present in both `personal/Paras` and `personal/Prutha` would be correctly detected, but the log output is misleading.

### P2-7 — Hygiene cleanup order: files deleted before index updated
**File**: `hygiene.py:39–44`
Files are deleted first, then document index entries are removed. If file deletion partially succeeds but index removal fails, the index contains entries for non-existent files.
**Fix**: Remove from index first, then delete files.

### P2-8 — SHA-256 of files <10KB skipped — misses small duplicate docs
**File**: `duplicate_scanner.py:40`
`_MIN_SIZE_BYTES = 10 * 1024` skips files under 10 KB. This excludes small text files, tiny PDFs, and thumbnails. For a document-focused NAS, 1 KB threshold is more appropriate. Consider lowering or making configurable.

### P2-9 — `file_sorter.py` — potential misclassification of video files
**File**: `file_sorter.py:129`
Document detection uses substring matching on filename stems (e.g., "invoice", "document", "receipt"). A video named `Documentary.mp4` would match "document" and be moved to Documents/. Use word-boundary matching or restrict to known document extensions.

### P2-10 — `store.py` — refresh tokens silently dropped on corruption
**File**: `store.py:405–407`
If `tokens.json` is corrupt, all refresh tokens are silently reset to `[]`. Users get logged out with no explanation. Should log a WARNING with details.

### P2-11 — Stale comment: nightly scan says "2:00 AM" runs at 4:00 AM
**File**: `main.py:330`
Comment is stale from when the scan time was changed from 2 AM to 4 AM. Minor but misleading.

### P2-12 — `index_watcher.py` persisted state uses `tuple()` unsafely
**File**: `index_watcher.py:127`
`_load_persisted_state()` calls `tuple(v)` on JSON values. If state file is corrupted with non-iterable values, this raises an unhandled exception that crashes the watcher entirely.
**Fix**: Wrap in try/except and reset to `{}` on error.
